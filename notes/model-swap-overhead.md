# oMLX 模型切換開銷研究

## 問題發現

在首次 benchmark 測試中，0.8B 模型回答簡單問題「2+2」竟然花了 **206 秒**，而 9B 只需 0.53 秒。

## 原因分析

oMLX 的 GPU 記憶體管理機制：
- 同一時間只有**一個模型**載入 GPU 記憶體（預設行為）
- 切換模型時需要：卸載當前模型 → 從磁碟載入新模型 → 初始化
- 這個載入過程是同步阻塞的

## 實測數據

### 模型切換延遲（未 pin 時）

| 模型 | 首次載入（冷啟動） | 已載入後 |
|------|-------------------|---------|
| Qwen3.5-0.8B（FP16） | **206.45s** | N/A |
| Qwen3.5-0.8B-MLX-4bit | **24.3s** ~ **122s** | **0.15s** |
| Qwen3.5-9B-MLX-4bit | **0.5s**（已在記憶體） | **0.53s** |

### 為什麼 FP16 比 4-bit 更慢？

| 模型 | 磁碟大小 | 載入時間 |
|------|---------|---------|
| Qwen3.5-0.8B（FP16） | 1.6 GB | ~206s |
| Qwen3.5-0.8B-MLX-4bit | 622 MB | ~24s |
| Qwen3.5-9B-MLX-4bit | 5.6 GB | ~1.5s（已快取） |

FP16 模型雖然只有 0.8B 參數，但檔案大小是 4-bit 版的 2.5 倍，載入時間反而更長。

## 解決方案：Model Pinning

### oMLX 多模型常駐機制

oMLX 支援三個關鍵功能：
1. **Model Pinning** — 釘選模型，永遠保持載入
2. **Per-model TTL** — 閒置超時自動卸載
3. **LRU Eviction** — 記憶體不夠時驅逐最久沒用的模型

### 配置

編輯 `~/.omlx/model_settings.json`：

```json
{
  "version": 1,
  "models": {
    "Qwen3.5-9B-MLX-4bit": {
      "force_sampling": false,
      "is_pinned": true,
      "is_default": false
    },
    "Qwen3.5-0.8B-MLX-4bit": {
      "force_sampling": false,
      "is_pinned": true,
      "is_default": false
    }
  }
}
```

### Pin 後的效果

| 步驟 | 時間 | 說明 |
|------|------|------|
| 0.8B 首次載入 | 69.7s | 冷啟動（重啟後首次） |
| 9B 呼叫 | 0.52s | 已 pin，沒被驅逐 ✅ |
| 0.8B 再次呼叫 | **0.14s** | 也留在記憶體 ✅ |

### 記憶體佔用

兩個模型同時 pin 的總記憶體：
- 0.8B-4bit: ~0.5 GB
- 9B-4bit: ~5.6 GB
- **合計: ~6.1 GB**（16GB RAM 完全夠用）

## 結論

1. **FP16 的 0.8B 沒有使用價值** — 已刪除，省 1.6GB 磁碟
2. **Pin 兩個模型** — 切換零延遲，0.8B 的速度優勢（6x）才能真正發揮
3. **oMLX 重啟後** — pinned 模型會在首次請求時自動載入，之後常駐
4. **autoRouteProfile 的實際延遲** — pin 前: 0.8B 路由要 24s + 9B 推理；pin 後: 0.8B 路由 0.1s + 9B 推理 0.5s
